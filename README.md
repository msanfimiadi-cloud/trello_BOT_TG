# Внутренний Telegram-бот для задач Trello

Бот принимает итоги встречи, предварительно показывает распознанные пункты, последовательно уточняет дедлайн и ответственного и создаёт карточку **только после явного подтверждения**. Прогресс хранится в SQLite и переживает перезапуск.

## Формат сообщения и сценарий

Первая непустая строка становится названием встречи. Поддерживаются маркеры `1.`, `1)`, `1 -` и `1.Задача`; строки до следующего маркера остаются частью текущей задачи.

```text
Пост-мит Авито 02.07

1. Подготовить отчёт по рекламным кампаниям
2. 16.07 — провести созвон и подвести итоги
3. Согласовать новый план публикаций
```

После подтверждения списка бот для каждого пункта показывает найденную дату (или просит её), предлагает ответственного из реально привязанных пользователей и выводит финальные название, описание, дедлайн и назначение. Ошибочный запрос можно повторить; пропуск и отмена доступны на каждом этапе. Поддерживаются `/start`, `/help`, `/cancel`, `/new`, `/status`, `/check_trello`, `/trello`, `/myid` и административная `/admin`.

## Получение настроек

1. Откройте **@BotFather**, выполните `/newbot` и сохраните выданный токен.
2. Получите API Key на странице `https://trello.com/power-ups/admin` и выпустите Token по ссылке авторизации Trello. Не отправляйте их в чат и Git.
3. ID списка можно получить, открыв `https://api.trello.com/1/boards/<BOARD_ID>/lists?key=<KEY>&token=<TOKEN>` локально, либо через инструменты разработчика Trello. Найдите нужное имя списка и его `id`.
4. Telegram user ID администратора можно узнать у служебного бота наподобие `@userinfobot` или через метод Bot API `getUpdates` до запуска polling. Запишите его в `ADMIN_TELEGRAM_USER_ID`.
5. Скопируйте шаблон и заполните значения:

```bash
cp .env.example .env
$EDITOR .env
```

Все перечисленные секреты и `ADMIN_TELEGRAM_USER_ID` обязательны. Доступ остальных пользователей хранится в SQLite и управляется через `/admin`, `/admin add <user_id>` и `/admin revoke <user_id>`; `/myid` доступна без авторизации. При первом запуске после обновления значения старой необязательной переменной `ALLOWED_TELEGRAM_USER_IDS` однократно импортируются в БД, после чего переменную следует удалить. `BOT_TIMEZONE` по умолчанию — `Europe/Moscow`.

Каждый допущенный пользователь выполняет `/trello <member_id>`. Бот получает участников настроенной доски через Trello API, принимает только существующий на ней Member ID и сохраняет ID вместе с актуальным отображаемым именем в SQLite. `/trello` без аргументов показывает текущую связь, повторная команда с ID изменяет её. Один Trello-участник не может быть связан с двумя Telegram-пользователями. Ответственные в карточках назначаются непосредственно по сохранённому `trello_member_id`; локальный `TEAM_JSON` больше не используется. При обновлении со старой версии существующий `TEAM_JSON` можно оставить ровно на один запуск: совпавшие по полному имени или username пользователи будут однократно перенесены, после чего переменную необходимо удалить.

Дедлайн отправляется в Trello как ISO datetime на **18:00 в `BOT_TIMEZONE`**: это не полночь, поэтому преобразование в UTC не сдвигает отображаемую календарную дату назад для типичных рабочих часовых поясов.

## Локальный запуск (Python 3.11+)

```bash
python3.11 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
set -a; source .env; set +a
python -m app.bot
```

Для разработки установите тестовые зависимости. Тесты не обращаются к настоящему Trello:

```bash
pip install -r requirements-dev.txt
pytest -q
```

## Docker

```bash
docker compose up --build -d
docker compose logs -f bot
```

SQLite размещается в именованном volume `bot-data`, контейнер автоматически перезапускается. Обновление:

```bash
git pull
docker compose up --build -d
docker image prune -f
```

Остановка без удаления данных: `docker compose down`. Удаление вместе с БД: `docker compose down -v`.

## Диагностика

- **Ошибка конфигурации** — проверьте наличие обязательных ключей и положительный целый `ADMIN_TELEGRAM_USER_ID`. Устаревший `TEAM_JSON`, если он оставлен только для миграции, должен быть валидным JSON-объектом.
- **401** — перевыпустите Trello Token и проверьте API Key; **403** — предоставьте пользователю доступ к доске; **404** — перепроверьте `TRELLO_LIST_ID`; **429** — подождите снятия лимита.
- `/check_trello` безопасно проверяет авторизацию и доступность списка.
- Если бот не отвечает, администратор должен проверить доступ через `/admin` и логи `docker compose logs -f bot`.
- При ручном имени Trello-участник не назначается: имя честно сохраняется только в описании.
- Каждая карточка получает стабильный системный ID задачи в описании. Перед созданием и ручным повтором бот ищет этот ID в целевом списке: если предыдущий запрос фактически успел создать карточку, сохранится найденная карточка, а дубль не появится.
- Если процесс остановился во время запроса создания, задача после запуска помечается ошибочной и **не повторяется автоматически**. Нажмите «Повторить»: сначала бот попытается восстановить результат по системному ID. Trello API не предоставляет настоящего идемпотентного ключа, поэтому при одновременном удалении/изменении маркера вручную абсолютная гарантия невозможна.
- Стартовая проверка не создаёт тестовую карточку (чтобы не засорять рабочий список): готовность к записи определяется доступностью незакрытого списка; окончательная проверка прав происходит при первом создании.

Логи не содержат токены или URL с query-параметрами. Резервируйте SQLite volume средствами вашей платформы.
